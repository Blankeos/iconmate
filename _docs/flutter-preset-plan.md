# Flutter Preset — Plan

Status: **Planning.** Reflects decisions from the initial design discussion. No code written yet.

## 1. Goal

A Flutter dev running iconmate in their project should get the same "add an icon and forget about it" experience the React/Solid/Svelte/Vue presets already give:

```bash
iconmate add --icon heroicons:heart --name heart
```

…drops the SVG into `assets/icons/`, adds a line to `icons.dart`, and the icon is usable as:

```dart
import 'package:my_app/assets/icons/icons.dart';

SvgPicture.asset(AppIcons.heart, width: 24);
```

No `flutter_gen`, no `pubspec.yaml` edits to babysit, no stringly-typed asset paths at call sites.

## 2. Design decisions (locked)

### 2.1 Preset

- New enum variant: `Preset::Flutter`.
- File extension on disk: `.svg` (same raw SVG as `normal` — no Dart source per icon).
- Default folder when preset is Flutter: `assets/icons` (no `src/` prefix).

### 2.2 Barrel file shape — constants only

Generate / maintain a Dart file like:

```dart
// GENERATED by iconmate — edits inside the class body may be lost.
class AppIcons {
  AppIcons._();
  static const String heart        = 'assets/icons/heart.svg';
  static const String chevronRight = 'assets/icons/chevron_right.svg';
}
```

- **No `flutter_svg` import in the barrel.** The barrel is pure string constants. The user imports `flutter_svg` in their own code and writes `SvgPicture.asset(AppIcons.heart, …)` at call sites. Keeps the barrel dependency-free and keeps iconmate out of the widget-API business.
- **No widget-factory variant** (e.g. `AppIcon.heart({width, height, …})`) in the initial release. Can be added later behind a flag if users ask.

### 2.3 Parse-and-edit, not regenerate-from-scratch

- The `.dart` barrel **is** the source of truth for identifier → path mappings (same role `index.ts` plays today for the JS presets).
- `add` / `delete` / `rename` parse the existing barrel, mutate the entry, write it back. Directly parallel to how the JS presets treat `index.ts`.
- **No sidecar JSON, no metadata file.** The barrel is self-contained state.
- `src/utils.rs:413` (`get_existing_icons`) already has a `FUTURE: flutter support` comment — this is the function that needs to branch on preset.

### 2.4 `--name` semantics — identical to JS presets

- `--name <identifier>` sets the Dart identifier that appears at call sites (`AppIcons.<identifier>`).
- When `--name` is omitted, infer from the SVG filename via the sanitization rules below.
- `--name` does **not** introduce a separate alias concept — it's the same thing as today, just lowerCamelCase instead of PascalCase.

### 2.5 Identifier sanitization

Applied when inferring an identifier from a filename, or validating one passed via `--name`:

1. Strip extension.
2. Normalize to `lowerCamelCase` — regardless of input casing. `snake_case`, `kebab-case`, `PascalCase`, and `SCREAMING_SNAKE` all collapse to lowerCamel. A literal `--name Heart` gets normalized to `heart` with a one-line notice (`Normalized "Heart" → "heart" (Dart convention).`).
3. If the result starts with a digit: prefix `icon` → `24_hours.svg` becomes `icon24Hours`.
4. If the result is a Dart reserved word (`class`, `new`, `void`, `switch`, …): suffix `Icon` → `class.svg` becomes `classIcon`.
5. If two files sanitize to the same identifier: **hard error**. Tell the user to rename one or pass `--name`. Don't auto-suffix `2`.

**Naming conventions — why lowerCamelCase is non-negotiable in Dart**

Unlike JS, Dart's style rules are enforced by lints that ship in every `flutter create` project (`flutter_lints` → `constant_identifier_names`). PascalCase is reserved for types (classes, enums, typedefs); lowerCamelCase is the rule for variables, functions, and `static const` fields. Writing `static const Heart = …` produces lint warnings at the definition and visually confuses readers (PascalCase signals "type" in Dart). The class wrapper (`AppIcons`) stays PascalCase because it's a class — so call sites read `AppIcons.heart`, which is exactly the `Type.member` shape Dart devs expect.

### 2.6 Project detection

- On startup, detect Flutter projects by the presence of `pubspec.yaml` containing a `flutter:` section.
- Parse `name:` from `pubspec.yaml` for the "paste this import into your Dart file" hint.
- When preset is unspecified and a Flutter project is detected, default to `flutter`. When preset is Flutter, default folder becomes `assets/icons`.

### 2.7 `pubspec.yaml` — do not auto-edit, do not parse

- iconmate **never writes to `pubspec.yaml`**. YAML is indentation-sensitive and full of user comments we shouldn't touch.
- v1 also does **not attempt to detect** whether the folder is already registered. Parsing YAML robustly enough to avoid false positives/negatives isn't worth the complexity for v1.
- Instead: on the first Flutter add per session, print an unconditional reminder ("Make sure `assets/icons/` is registered under `flutter: assets:` in pubspec.yaml.") once. Suppressible via `--no-pubspec-hint`.
- Smarter detection is a possible follow-up, not a blocker.

### 2.8 Barrel location — `lib/icons.dart` by default

Unlike the JS presets, the Flutter barrel does **not** sit inside the SVG folder. Dart's `package:` import URIs only resolve to files under `lib/`, so a barrel at `assets/icons/icons.dart` would force call sites into ugly relative imports (`import '../../assets/icons/icons.dart';`). The default avoids that:

- Default barrel path: **`lib/icons.dart`** (project-root-relative).
- Imported at call sites as: `import 'package:<pubspec_name>/icons.dart';`.
- The SVG folder (`--folder`) and the barrel location are **two independent paths**.

This is the one place where the Flutter preset structurally differs from the JS presets. Everything else (parse-and-edit, `--name` semantics, etc.) stays consistent.

### 2.9 Paths stored inside the barrel — project-root-relative

Each `static const` entry stores the SVG path as a string passed to `SvgPicture.asset(...)` at runtime. Flutter's asset loader resolves those strings against `pubspec.yaml` asset roots, which are **project-root-relative**. So:

```dart
static const heart = 'assets/icons/heart.svg';  // not '../assets/icons/heart.svg'
```

`write_dart_barrel` takes the project root and the SVG folder as inputs and emits root-relative strings, regardless of where the barrel file itself lives.

### 2.10 Configurable barrel file & class name

Default hardcoded, with two optional config keys:

```jsonc
{
  "preset": "flutter",
  "flutter_barrel_file": "lib/icons.dart",   // optional, default: lib/icons.dart
  "flutter_barrel_class": "AppIcons",        // optional, default: AppIcons
}
```

`flutter_barrel_file` is **project-root-relative** and **independent of `--folder`** (not a filename-within-folder like it would be in the JS presets). Examples a user might set:

- `lib/gen/icons.dart` — flutter_gen-style segregated codegen folder.
- `lib/assets/icons/icons.dart` — if they want co-location (requires moving SVGs under `lib/` too, and accepts that the `.dart` file gets bundled as an asset).
- `lib/src/constants/icons.dart` — matches their existing constants layout.

The class name shows up at every call site in the user's codebase — we shouldn't force a rename on projects that already have an `AppIcons` or want `Assets.icons` style.

### 2.11 File ownership — iconmate owns `icons.dart` entirely

Divergence from the JS presets, which do surgical edits to `index.ts` so users can hand-add their own exports.

- The Flutter barrel file is **fully owned by iconmate**. Every write reformats the whole file from the in-memory set of entries.
- The generated file opens with a `// GENERATED by iconmate — do not edit by hand.` header so readers know the rules.
- JS presets stay surgical as they are today. No change there — users can continue to hand-add exports to `index.ts`.
- Why the divergence: Dart's const-field grammar has too many shapes (`static const foo = '…'` vs. `static const String foo = '…'`, dartfmt line-wrapping, trailing commas, explicit types). Parsing arbitrary user-edited Dart defeats the purpose. Owning the file keeps parsing trivial (we only parse what we emit) and makes writes deterministic.

### 2.12 Rename semantics — iconmate owns the filesystem, LSP owns the symbol

`iconmate rename` for Flutter:

- **Renames the SVG file on disk** (e.g. `heart.svg` → `redheart.svg`).
- **Updates the path string inside the barrel** (e.g. `'assets/icons/heart.svg'` → `'assets/icons/redheart.svg'`).
- **Does NOT rename the Dart identifier.** The `static const heart` stays `heart`.

Result afterward:
```dart
static const heart = 'assets/icons/redheart.svg';
```

The identifier/filename mismatch is intentional. Rationale: renaming a Dart identifier means updating every call site across the user's codebase, which the LSP (`Rename Symbol` in VS Code, `Refactor → Rename` in IntelliJ) already does correctly. iconmate has no business AST-rewriting user code. Clean division: **iconmate owns the filesystem, the LSP owns the symbol table.** Document this behavior in the `rename` hint text so the UX is clear.

### 2.13 `output_line_template` interaction

`output_line_template` is meaningless when preset is Flutter (barrel format is fixed). Handling:

- If `output_line_template` is unset or equal to the default string: silently ignored when preset is Flutter.
- If `output_line_template` is set to a non-default (user-customized) value: **warn once at startup** — "`output_line_template` is ignored for the Flutter preset."

No silent footguns — if they bothered to customize it, they should know it's being dropped.

### 2.14 First-run barrel creation

- First `iconmate add` in a Flutter project when `lib/icons.dart` (or configured `flutter_barrel_file`) doesn't exist: create it with the generated header + the first entry.
- `lib/` exists in every `flutter create` project, so no `mkdir -p` for the default path.
- For custom paths like `lib/gen/icons.dart`: silently `mkdir -p` the parent dir.

### 2.15 Explicitly out of scope for v1

- Widget-factory barrel style (Option B).
- `vector_graphics_compiler` / `.vec` precompilation.
- Icon-font generation.
- Auto-editing `pubspec.yaml`.
- `iconmate flutter doctor` validation command.
- Hand-dropped SVG reconciliation — that belongs to `iconmate sync` (see `iconmate-sync-plan.md`).

## 3. Code touch points

Rough orientation for implementation. Not a task list.

  - Add `Preset::Flutter` to the enum + `PRESETS_OPTIONS` + `filename_from_preset` (returns `svg`).
  - `get_existing_icons` (line 413): branch on preset — parse `icons.dart` const declarations when Flutter, else parse `index.ts` exports.
- `src/config.rs`
  - Preset enum validation is automatic once the new variant is added.
  - `DEFAULT_FOLDER` becomes preset-aware (function, not const): `assets/icons` for Flutter, `src/assets/icons` otherwise.
  - Add `flutter_barrel_file` (default `lib/icons.dart`) and `flutter_barrel_class` (default `AppIcons`) config keys with allow-unknown-keys updates.
- New: `src/flutter.rs`
  - `parse_dart_barrel(barrel_path) -> Vec<(identifier, root_relative_path)>`.
  - `write_dart_barrel(barrel_path, class_name, entries)`.
  - `sanitize_identifier(raw) -> Result<String, CollisionOrInvalid>`.
  - `detect_flutter_project() -> Option<FlutterInfo { package_name }>`. Parses only `name:` from `pubspec.yaml`; no asset-list parsing in v1.
  - Barrel path and SVG folder are tracked as **separate inputs** throughout — the Flutter preset can't assume the barrel lives inside the icon folder.

  - Route through the Dart barrel parse/write path when preset is Flutter.

For Flutter, render `usage: AppIcons.{normalizedName}` using the configured `flutter_barrel_class`. Preview the lowerCamelCase-normalized name live as the user types so they see the exact call-site identifier they'll get.
  - Extract a small `usage_hint(preset, raw_name, class_name)` helper so this stays tidy and other preset-specific labels can flow through the same function later.
- `config-gen/src/schema.ts` + `types.ts`
  - Add `"flutter"` to the preset enum. Add the two new optional config keys. Regenerate via `just config-schema`.
- `README.md`
  - New row in the preset table. New section covering: install `flutter_svg`, register the folder in `pubspec.yaml`, then use `AppIcons.foo`.
- `skills/iconmate/SKILL.md`
  - Document the Flutter preset so agents know it's available.

## 4. Suggested build order

1. `parse_dart_barrel` + `write_dart_barrel` + `sanitize_identifier`, all pure functions, fully unit-tested.
2. `Preset::Flutter` enum + config schema + preset-aware `DEFAULT_FOLDER`.
3. Wire Flutter branching into `add` / `delete` / `rename` flows.
4. `pubspec.yaml` detection: package name + assets-registered check.
5. CLI hint for unregistered folders, with `--no-pubspec-hint` escape hatch.
6. README + skill docs.

Each step is independently shippable. (1) and (2) can land in the same PR.
